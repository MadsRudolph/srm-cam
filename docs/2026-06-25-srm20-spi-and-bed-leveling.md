# Dev log — double-sided dowels, bed leveling & live SRM-20 SPI control

**Date:** 2026-06-25
**Scope:** dialing in the double-sided dowel workflow, building manual + automatic
bed leveling, and discovering/driving the SRM-20's hidden SPI remote interface.

This is a detailed, feature-by-feature record (more than a changelog) so anyone —
human or AI — picking this up later inherits the *why* and the hard-won
discoveries, not just the diffs. Persistent machine facts also live in the
project memory (`srm20-*` notes).

---

## 1. Double-sided dowel workflow

### 1.1 Dowel test coupon (`examples/dowel_test/`)
A minimal double-sided board — **2 traces + 4 holes** + an asymmetric witness
mark — to verify the fresh-milled dowel registration end to end. Generated by
[`gerber2rml/examples/dowel_test.py`](../gerber2rml/examples/dowel_test.py). It's
a real gerber folder (not raw G-code) so it loads in the GUI and can be placed on
the bed. The `build_double_sided` flow adds the two dowel/align holes in the waste.

### 1.2 Per-pin dowel-hole clearance (kerf compensation) — a key discovery
**Interpolated (circled-out) holes on the SRM-20 mill UNDERSIZE.** First measured
~0.4 mm under nominal (3.1 mm commanded → 2.7 mm; 1.9 mm → 1.5 mm). We added a
`pin_clearance` that's added 1:1 to the milled hole diameter, then dialed it in
empirically with a swept fit-test coupon (§1.4). Result:

- **Big pin (3.1 mm): +0.20 mm clearance** (seats perfectly)
- **Small pin (1.9 mm): +0.15 mm clearance** (a touch tighter)

The kerf is **not identical at both diameters**, so clearance is **per-pin**
(`DowelSpec.clearance_large` / `clearance_small`), not one shared value. Dowel
centres are invariant to clearance, so the align-only re-cut lands back on the
existing holes.

### 1.3 Deeper bed bite (`DOWEL_BED_DEPTH`)
The fresh dowel hole must reach a fixed distance *below the stock* into the
sacrificial bed so the pin holds. Set to **5 mm** (align depth = stock thickness +
5 mm), after earlier 3 mm only reached ~2 mm in practice. If it still falls short,
the work surface may be near the bottom of the SRM-20 Z travel (see the
Z-travel-range memory) — lower it rather than just bumping the number.

### 1.4 Dowel-pin fit-test coupon (`examples/hole_test/`)
A **drill gerber** with two rows of holes at swept diameters — top row brackets
the small pin, bottom row the big pin — so you cut once and pick the snug hole.
This is how the §1.2 clearances were dialed.

### 1.5 Single-bit drilling is now the default
We almost always run one bit. `DrillJob.single_bit` now defaults to **True**
(plunge holes that fit, interpolate the larger ones into one file). Multi-bit
(one file per diameter) is opt-in via the CLI `--multi-bit` flag.

## 2. Bed leveling — the software half

**Problem:** an uneven bed means traces don't cut to depth — the centre may not
clear copper while the corners cut too deep. (A real board measured **130 µm**
of bow.)

### 2.1 Test board (`examples/level_test/`)
An 80 × 60 mm board with copper reaching all four corners + a centre cross, so
depth variation from tilt/warp shows up everywhere.

### 2.2 Height-map engine ([`gerber2rml/engine/leveling.py`](../gerber2rml/engine/leveling.py))
- `HeightMap` — **plane fit** (≥3 pts, corrects tilt) or **bilinear over a grid**
  (corrects bow); `from_points` auto-picks.
- `apply_leveling(paths, hmap)` — adds `hmap(x, y)` to every cut's Z **and
  subdivides long feed moves** so Z ramps along the warp, not just at endpoints.
  Runs LAST (after placement), in machine coordinates.
- Threaded into `build_jobs` / `state.export` via an optional `level` callable.

### 2.3 Manual probe path
`probe_points` lays out a grid; `write_probe_files` emits one tiny G-code program
per point + a checklist. **VPanel gotcha:** queuing them to pause between points
pops a full-screen modal that hides the Z readout — so this manual path is
clunky. That pain is what drove the SPI discovery below.

## 3. The SRM-20 SPI remote interface — the big discovery

The SRM-20 has an **official SPI remote header behind the back cover** (Roland
built it for exactly this). An Arduino Uno plugs into a shield there and drives
the machine. This unlocks live position readout and automatic probing — things
VPanel won't give us.

### 3.1 Library + hardware
- **Library:** the official Roland `SRM20SPIRemote` (©2014 Roland DG), publicly
  mirrored at **`github.com/shohei/srm20-arduino`** (NOT in `IAMAS/monoFab`, which
  only has the iBeacon/exterior demos). Vendored at
  [`hardware/SRM20SPIRemote/`](../hardware/SRM20SPIRemote/).
- **Shield:** `shohei/srm20_spindle_shield` (Uno footprint; SPI rides the ICSP
  header, so it's board-independent).
- **Pins:** `begin(9, 6)` — slave-select D9, ready D6 (Roland's examples; NOT the
  `begin(9,2)` from the chocolate-milling hack).
- **Units: MICRONS** (1/1000 mm) — confirmed in Roland's `SRMTest.ino`.
- **Use an Uno**, not a Mega (Mega's hardware SPI is on 50–52, not 11–13; ICSP
  avoids that, but Uno is the tested path).

### 3.2 What works vs. what doesn't (validated on the machine)
- ✅ **`jumpTo`** — moves reliably, including Z (one online report claimed Z
  "wouldn't move"; on this machine it does).
- ✅ **`getActualPosition`** — reads true position.
- ⚠️ **SPI reads are FLAKY** — a single read can return garbage (e.g. `0,0,0`,
  which once sent a `jumpTo` to machine home/top). **Never move on one read.**
  The fix: read twice and require agreement (`readPos`).
- ❌ **`readSensor` / `scanTo` / `getScanPosition` — UNUSABLE.** `readSensor`
  returns `0` then `64` on two back-to-back calls with nothing touching (SPI
  framing garbage), its "latch" never clears, and `scanTo` produced no motion.
  The native contact-scan path was abandoned.
- ✅ **VPanel coexists** with the Arduino on the SPI link **and mirrors its
  moves** — so you never swap cables: probe with the Arduino, cut with VPanel.

### 3.3 The probe that actually works — EXTERNAL touch probe
Because the machine's own sensor is unreliable over SPI, we sense contact with
our own circuit on an Arduino pin (no SPI involved):

- **Wiring:** the copper board, **isolated from the bed** (paper/tape underneath),
  → **D7** (floats HIGH via `INPUT_PULLUP`); the tool/collet → **Arduino GND**.
  The tool is already frame-ground (tied to Arduino GND through the shield), so
  the *floating copper* must be the sensed side. Tool touches copper → D7 LOW.
- **Probe motion:** step Z down 25 µm via `jumpTo`, check D7, on contact read Z.
  **Repeatable to the step** (tested: −56.290 mm every run).
- **Spindle stays OFF** the whole time. No clip touches a spinning bit.

### 3.4 Coordinate frames — tested, important
- **`getActualPosition` reads in the G54 / user coordinate system.** Setting the
  G54 origin "here" in VPanel made X/Y read `0,0`.
- **`setOrigin`/`getOrigin` is a SEPARATE origin** — it did NOT change when G54
  was set, so **you cannot set the G54 cut datum via SPI `setOrigin`.** Datum
  workflow: "Probe Z" auto-finds the surface, then set G54 Z "here" in VPanel.
- Consequence: the live tool overlay/jog (DRO frame = G54, preview frame =
  bed/machine) line up **only when G54 is at machine home**. A click-on-preview
  "Sync" to capture the offset was tried and **rejected as too imprecise**. Bed
  leveling is unaffected (it's datum-relative).

## 4. Live machine GUI

A top **machine bar** + preview overlays, all off the SPI link:

- **DRO banner** — live X/Y/Z (G54) at ~4 Hz, with a **garbage-jump filter**
  (rejects >40 mm spikes, re-syncs after a few) so jogging shows smoothly but SPI
  glitches don't fling the readout.
- **Tool marker** on the preview — green crosshair+ring tracking the spindle;
  **turns red on contact**.
- **Touch indicator** — `bit ○ clear` / `bit ● TOUCHING` (the external probe).
- **Click to jog** — click a point, the tool **lifts ~5 mm then travels XY** there.
- **Probe Z** — auto touch-off: descends from here until contact, stops at the
  surface, sets that as the work-surface zero (DRO then shows depth-below-surface).
- **Probe over SPI** (bed leveling) — sends the grid to the Arduino, auto-probes
  each point, fills the leveling table, then warps the export.
- **Height map** views — a 2D heatmap overlay on the preview and an **OctoPrint-
  style 3D bed visualizer** ([`gui/bedviz.py`](../gerber2rml/gui/bedviz.py),
  pyqtgraph GL) with Z exaggeration.

Probing/touch-off/jog share one serial connection; the DRO poll pauses for them
and resumes.

## 5. Hardware / firmware reference

- [`hardware/SRM20SPIRemote/`](../hardware/SRM20SPIRemote/) — vendored Roland library.
- [`hardware/srm20_spi_validate/`](../hardware/srm20_spi_validate/) — bench sketch
  used to discover everything (`p` position, `u/d` Z nudge, `s` sensor, `b`
  single-point probe, etc.).
- [`hardware/srm20_spi_probe/`](../hardware/srm20_spi_probe/) — the production
  sketch the GUI drives. Serial protocol (115200 baud, microns):
  - `D` set datum · `P id x y` probe point → `R id x y z` · `Q` fast position+touch
    → `Q x y z t` · `J x y` jog (lifts then travels) · `T` touch-off → `T x y z` ·
    `G` get origin · `O`/`R` setOrigin test/restore.

**Serial host driver:** [`gerber2rml/engine/spi_probe.py`](../gerber2rml/engine/spi_probe.py)
(`probe_grid`, `query_position`, `jog_to`, `touch_off`, `open_link`). Note the
**Uno auto-resets when the port opens** — the driver waits ~2 s and retries the
datum handshake.

## Emergency STOP / abort (added after a near-miss)

A probe run drove the bit toward the bed; opening the lid paused the machine, but
the firmware kept *queuing deeper Z moves* (its `waitForMotorStop` couldn't tell
"done" from "paused"), so closing the lid resumed the descent. Disconnecting the
GUI did nothing because the Arduino probes autonomously. Fixes, three layers:

- **Firmware (`srm20_spi_probe.ino`) — self-protecting.** `waitForMotorStop` now
  returns false if a move doesn't complete in time (i.e. the machine is paused)
  **or** an `!` abort byte arrives. The probe/touch-off loops check it between
  every 25 µm step and, on false, **lift to safe Z and stop** instead of stepping
  again. New top-level `!` command also lifts. **You must reflash the Arduino**
  with this sketch for the STOP button to fully work.
- **Driver (`spi_probe.py`).** `probe_grid`/`touch_off` take `should_abort`; on
  abort they send `!` (firmware lifts) and bail. Serial opens with a short read
  timeout so a STOP is responsive mid-read.
- **GUI.** A red **STOP** button (always enabled) aborts probing/touch-off/jog,
  lifts the tool, and disconnects. Plain **Disconnect** now also aborts a running
  probe. NOTE: this is a software backstop — the machine's own lid interlock /
  power switch is still the real emergency stop.

**Automatic runaway guard (outlier Z).** A bed's surface is flat to a fraction of
a millimetre, so a probe that comes back much deeper than the others didn't find
copper — the bit is heading into the board/bed. Both layers now catch this:

- **Firmware**: the first good touch sets a reference surface; later points may
  not descend more than `OUTLIER_MARGIN_UM` (1.2 mm) past it without contact. Hit
  that limit → lift and report `E <id> RUNAWAY` instead of plunging to the 6 mm
  cap. (Tilt/warp is far less than 1.2 mm, so real points still touch first.)
- **Host** (`probe_grid`, `outlier_mm=1.5`): a firmware `RUNAWAY`, or any touch
  more than `outlier_mm` deeper than the first point, aborts the whole grid —
  lifts (`!`), stops, and the GUI shows which point ran away.

## Hard-won gotchas (read before touching the SPI path)
- **Never trust a single SPI read** to drive motion — require two agreeing reads.
- **`readSensor`/`scanTo` are unreliable** — use the external D7 touch probe.
- **The Uno resets on port open** — wait for it; only one program can hold the COM
  port (close the Arduino Serial Monitor before the GUI grabs it).
- **The probed copper must be isolated from the bed** or D7 reads "touched"
  forever (the bed is grounded to Arduino GND through the shield).
- **`setOrigin` ≠ G54** — can't set the cut datum from the GUI; zero in VPanel.
- **VPanel + Arduino coexist** — no cable swapping.
- **Measure, don't assume:** kerf clearance, bed-bite depth, and every SPI command
  were dialed/validated on the actual machine, not from datasheets.
