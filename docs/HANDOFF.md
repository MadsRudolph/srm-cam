> **2026-06-30 — spindle spin-up settle + ramped lead-in.** NC header dwells
> (`G04 X<sec>`, default 2 s) after `M3` so the spindle is at full RPM before the
> first cut (SRM-20 has no programmable spindle speed — RPM is a VPanel setting).
> Cut paths (traces/cut-out) enter via a ramped lead-in (`engine/leadin.py`,
> `apply_lead_in`) not a vertical plunge; drills stay vertical. On by default
> (`build_jobs(..., lead_in=)`, `gcode.render(..., spinup_s=)`). Details:
> `docs/2026-06-30-spindle-spinup-and-lead-in.md`.

> **2026-06-30 — V-bit engraving support added.** Trace isolation now supports a
> V-shaped engraving bit (for ~0.2 mm SMD traces) next to the flat endmill. Cut
> width is depth-dependent: `W = T + 2*D*tan(theta/2)`, driven width-first
> (`TraceJob.tool_type="vbit"`, set `target_width`, depth is derived). `isolate()`
> uses `TraceJob.effective_diameter()/effective_cut_depth()`; pre-flight warns on a
> V-bit without bed leveling. Details: `docs/2026-06-30-vbit-engraving-support.md`.

## ▶ NEXT TASK: double-sided-aware preview
**Problem:** with the **Double-sided** checkbox on, the preview still shows the *single-sided* board (`state.board.holes`) and **does NOT show the dowel/alignment holes** or the reflected top side. The pins are only created at export time inside `build_double_sided`. User wants the preview to **show both sides of the PCB and the dowel placement** when Double-sided is ticked.

**Where:** `MainWindow.generate_preview()` in `gerber2rml/gui/app.py` (currently: for the drill tab it overlays `state.board.holes` on the trace context; otherwise shows the op's toolpaths). Canvas helpers in `gui/canvas.py`: `show_segments(cuts, rapids, holes=)`, `show_holes`, `show_gaps`. Toolpath→polyline via `app/preview.toolpath_segments`. The double-sided geometry comes from `doublesided.layout_double_sided(self.state.gerber_dir, ...)` (reloads from disk; returns shapely `bottom_copper`, `top_copper`, `outline`, `holes`, `align_holes` [(x,y,d)], `y_axis`).

**Suggested approach (confirm with user):** when `self.double_sided_chk.isChecked()` and a board is loaded, `generate_preview` computes `lay = layout_double_sided(self.state.gerber_dir, ...)` and previews from the **double-sided frame** instead of `state.board`:
- Draw **bottom** isolation (cyan, from `isolate(lay.bottom_copper, trace, outline=lay.outline)`) and **top** isolation (a 2nd distinct colour, e.g. magenta, from `isolate(lay.top_copper, trace, outline=_reflect_geom(lay.outline, lay.y_axis))`) — overlay both so you see both sides registered.
- Draw the board holes (`lay.holes`) red + the **two dowel holes** (`lay.align_holes`) as a clearly distinct marker (e.g. large yellow ⊕ / circle) so placement is obvious.
- The cutout tab can show `lay.outline`.
- Keep the single-sided path unchanged when the box is off.
Add an offscreen test in `tests/test_window.py` (build MainWindow, load FIXT, check `double_sided_chk`, `generate_preview()`, assert the dowel markers are drawn — e.g. canvas has ≥2 extra patches/scatter for the align holes). Possibly add a `PreviewCanvas` method like `show_pins(align_holes)` for the distinct dowel markers. Consider caching the layout so live-edit previews aren't slow (layout reloads from disk each call).

**Gotcha:** the double-sided frame is shifted (pins beyond the 104 box → board no longer at origin); the view must autoscale to include the pins. Use `tests/fixtures/mosfet_test` (a ~104×104 board WITH F.Cu) and `examples/calibration` for testing.
